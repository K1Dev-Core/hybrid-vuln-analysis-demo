## สรุปสั้น ๆ

ระบบมองว่าไฟล์ `command_injection_challenge.c` มี data flow ที่เสี่ยงจาก `fgets` ไปหา `system` และ pattern นี้เข้าข่าย command injection ค่อนข้างชัด

## ต้นเหตุของปัญหา

input จากผู้ใช้ถูกเก็บไว้ใน `user_input` แล้วถูกเอาไปประกอบเป็นคำสั่งผ่าน `snprintf` ก่อนส่งต่อให้ `system()` รันอีกที  
ต่อให้มีการเช็กเงื่อนไขบางอย่างอยู่บ้าง แต่ถ้าไม่ได้คุมค่าที่รับเข้ามาแบบเข้มจริง ก็ยังเสี่ยงอยู่ดี

## หลักฐานที่ระบบเจอ

- ตัววิเคราะห์ที่ใช้คือ `joern-cpg`
- source คือ `fgets` ที่บรรทัด 29
- sink คือ `system` ที่บรรทัด 45
- คะแนนจาก model อยู่ที่ `0.075`
- `Joern` บอกว่าจุดนี้อยู่ใน method `m`

ช่วงโค้ดที่น่าสงสัย:

```text
0035:     if (strlen(user_input) == 0) {
0036:         puts("empty input");
0037:         return 1;
0038:     }
0039:
0040:     if (strcmp(user_input, "admin") == 0) {
0041:         puts("hello admin");
0042:     }
0043:
0044:     snprintf(command, sizeof(command), "echo hello %s", user_input);
0045:     system(command);
0046:     return 0;
0047: }
```

## ความรู้ที่ดึงมาประกอบ

- เอกสาร `command_injection.md` ถูกดึงขึ้นมาแรงสุด เพราะ pattern ตรงกับเคสรับ input แล้วไปจบที่ `system`
- เอกสาร `format_string.md` ถูกดึงขึ้นมาประกอบเล็กน้อย เพราะมีตระกูล `printf/snprintf` อยู่ใน flow

## แนวทางแก้

- ถ้าเลี่ยง `system()` ได้ ควรเลี่ยงก่อน
- ถ้าจำเป็นต้องประกอบคำสั่งจริง ๆ ให้ใช้ allow-list ของค่าที่รับได้
- อย่าเอา input ดิบจากผู้ใช้ไปต่อ string แล้วรันทันที
- ถ้าค่าที่รับเข้ามาควรเป็นแค่ชื่อหรือ token สั้น ๆ ให้คุมรูปแบบตั้งแต่ต้นทางเลย

## ระดับความมั่นใจ

ค่อนข้างมั่นใจว่าจุดนี้เสี่ยงจริง เพราะทั้ง static path, binary evidence, และลักษณะโค้ดไปทางเดียวกัน  
แต่ก่อนสรุปเป็นช่องโหว่จริงในงาน production ก็ควรมีคน review ซ้ำอีกครั้ง
